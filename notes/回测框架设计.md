# 回测框架设计

> 回测 = 用历史数据模拟策略在过去的表现。核心原则：**回测环境必须尽可能接近真实交易环境**。

---

## 一、回测的两种范式

### 1. 向量化回测 (Vectorized)

对整个时间序列一次性计算，用矩阵/向量运算。

```
信号 = 计算因子(历史OHLCV)      # [T x 1] 向量
持仓 = 信号转仓位(信号)          # [T x 1]
收益 = 持仓 * 价格变化            # [T x 1] 逐元素
累计PnL = cumsum(收益)
```

| 优点 | 缺点 |
|------|------|
| 极快（numpy 向量化） | 无法建模路径依赖 |
| 代码简洁 | 难以处理限价单/部分成交 |
| 适合因子研究阶段 | 假设流动性无限 |

**适用场景**: run4 的因子质量评估（只需算 IC/ICIR/R²，不需要精确回测）。

### 2. 事件驱动回测 (Event-Driven)

按时间顺序逐分钟/逐笔模拟：数据到达 → 生成信号 → 下单 → 撮合 → 更新持仓。

```
for t in timestamps:
    bar = data[t]                    # 新数据到达
    signal = strategy.on_bar(bar)    # 策略生成信号
    orders = signal_to_orders(signal) # 信号转订单
    fills = matcher.match(orders, bar) # 撮合引擎
    portfolio.update(fills)           # 更新持仓
    recorder.log(t, portfolio)        # 记录状态
```

| 优点 | 缺点 |
|------|------|
| 真实模拟交易流程 | 实现复杂 |
| 支持限价单/止损单 | 速度慢（O(n²) 撮合） |
| 可引入冲击模型 | 需要更多参数 |

**适用场景**: run5 的完整策略回测（限价单 + 冲击模型）。

## 二、回测框架的核心模块

### 1. 数据模块

```
数据来源 → 清洗 → 对齐时间戳 → 存储
```

- **OHLCV**: Open, High, Low, Close, Volume（分钟/日级）
- **订单簿数据**: L1（最优买卖价+量）/ L2（多档深度）
- **数据质量检查**: 缺失值、跳价、时间戳不一致、涨跌停/异常价格
- **数据分割**: 训练集/验证集/测试集，绝不能交叉

**我的经验**: MadEvolve 复现中用了 Binance BTCUSDT 替代 Polygon BTCUSD。两个数据源的交易时间、节假日、最小价格变动单位都不同，需要对时间戳对齐全量检查。

### 2. 策略模块

```
输入: 当前 Bar / 历史数据
输出: 目标仓位 [-1, 1] 或 订单列表
```

关键接口：

| 函数 | 作用 | MadEvolve 对应 |
|------|------|----------------|
| `set_target(state)` | 计算目标仓位 | `EVOLVE-BLOCK-TARGET` |
| `set_limit_order(state, target)` | 生成限价单 | `EVOLVE-BLOCK-LIMIT` |
| `default_calcset(data)` | 因子计算（特征工程） | `EVOLVE-BLOCK-CALCSET` |

**设计要点**:
- 策略只能看到**当前时刻及之前**的数据（禁止未来函数）
- 状态管理要清晰：当前持仓、现金、未成交订单
- 避免在策略内部硬编码交易参数

### 3. 撮合引擎

```
订单 → 撮合规则 → 成交列表
```

**限价单撮合逻辑**（MadEvolve 的实现）：

```
给定: 订单价格 P_order, 订单方向 dir (买/卖), 当前 Bar 的 OHLCV

if dir == BUY:
    if P_order >= Low:    # 价格触及
        成交价 = min(P_order, Open)  # 保守估计：用开盘价
        hit_ratio = 1.0              # 假设全部成交
else:
    if P_order <= High:
        成交价 = max(P_order, Open)
        hit_ratio = 1.0
```

**现实比这复杂得多**:
- 实际挂单可能部分成交
- 大单会有滑点（吃的价位越来越差）
- 排队顺序影响成交概率
- 撤单/改单逻辑

### 4. 冲击模型 (Market Impact)

> 每次交易都会推动价格向不利方向移动。

MadEvolve 使用的是 **6 参数平方根冲击模型**:

```
瞬时冲击 = σ * a * (Q / V)^b          # 立刻发生
永久冲击 = σ * c * |Q / V|^d * sign(Q) # 信息泄露
衰减函数 G(τ) = e^(-τ / τ_0)          # 指数衰减
总冲击 = 瞬时冲击 * G(τ) + 永久冲击   # 随时间衰减
```

参数说明:

| 参数 | 含义 | 典型值 |
|------|------|--------|
| σ | 波动率 | 从数据估计 |
| a | 瞬时冲击系数 | ~0.1 |
| b | 瞬时冲击指数 | ~0.5 (平方根定律) |
| c | 永久冲击系数 | ~0.01 |
| d | 永久冲击指数 | ~0.5 |
| τ_0 | 衰减时间常数 | ~1 分钟 |

**我的经验**: 冲击模型的计算是 O(n²) 的（每笔交易要累加之前所有交易衰减后的冲击）。MadEvolve 复现中用 numpy 向量化 + 预计算衰减查表 + 滑动窗口裁剪，把 42 分钟压缩到 7 分钟。

### 5. 绩效评估模块

**核心指标**:

| 指标 | 公式 | 含义 |
|------|------|------|
| PnL | Σ(成交收益 - 手续费 - 冲击成本) | 总盈亏 |
| 冲击调整 PnL | PnL - 冲击成本 | 扣除市场影响后的真实利润 |
| Sharpe | (年化收益 - 无风险利率) / 年化波动率 | 风险调整收益 |
| Calmar | 年化收益 / 最大回撤 | 回撤调整收益 |
| 最大回撤 | max(峰值 - 谷值) / 峰值 | 最坏情况亏损 |
| 胜率 | 盈利次数 / 总交易次数 | 交易胜率 |
| 单笔最大盈利/亏损 | max/min(单笔 PnL) | 尾部风险 |

**为什么 MadEvolve 选 PnL 而非 Sharpe 作为适应度**:

Sharpe 可以通过"选择性交易"（只在确定性高时交易，减少波动率分母）来人为刷高。冲击调整 PnL 内含成本惩罚——做大单会被冲击成本惩罚，做高频会被手续费惩罚。这是更诚实的适应度函数。

## 三、回测的常见陷阱

### 1. 前视偏差 (Look-Ahead Bias)

在时刻 t 使用了 t+1 或之后的信息。

| 例子 | 检查方法 |
|------|----------|
| 用全样本做标准化再回测 | 必须用 expanding window |
| 信号用了 t+1 的价格 | 逐条检查时间戳 |
| 因子计算用了未来数据 | 对齐时间索引 |

### 2. 生存偏差 (Survivorship Bias)

只用了"现在还活着"的资产回测。退市的资产往往表现差，剔除它们会高估收益。

### 3. 过拟合 (Overfitting)

| 类型 | 表现 | 检测 |
|------|------|------|
| 参数过拟合 | 训练集超好，测试集崩 | IS-OOS 对比 |
| 结构过拟合 | 试了 100 种策略选最好的 | 多重检验校正 |
| 数据窥探 | 在测试集上调过参 | 严格锁定测试集 |

### 4. 交易成本低估

- **手续费**: Maker ~0.02%, Taker ~0.04%（币安现货）
- **滑点**: 市价单实际成交价偏离预期
- **冲击成本**: 大单推动市场（前面已讨论）
- **机会成本**: 资金占用期间的利息损失

## 四、回测 → 实盘的距离

| 回测假设 | 实盘现实 |
|----------|----------|
| 历史数据完整准确 | 数据有缺失、错误、延迟 |
| 订单立即成交 | 网络延迟、撮合排队 |
| 无对手方风险 | 交易所可能宕机/限频 |
| 无监管限制 | 持仓上限、KYC、税务 |
| 资金无限可分 | 最小下单量、最小价格变动 |

## 五、我的实践经验

从 MadEvolve 复现中总结的回测开发流程：

1. **先跑通向量化版本**——验证策略逻辑，秒级出结果
2. **加入交易成本**——手续费 + 简单滑点模型
3. **换事件驱动框架**——模拟逐分钟决策
4. **加入冲击模型**——评估大仓位策略的真实性
5. **性能优化**——向量化、查表、窗口裁剪
6. **交叉验证**——不同时间段、不同参数、不同随机种子

**一个实用技巧**: 写回测时先不做优化，让代码清晰可读。跑通后再做性能优化。正确的慢回测 > 错误但快的回测。
