# 回测框架设计：从零搭建一个不会骗你的回测系统

> 一份"写给别人的笔记"——从回测的本质讲起，到架构设计、陷阱避坑、开源框架对比，再到实盘衔接。读完你应该有能力设计自己的回测引擎，或者至少知道现有的引擎在什么地方可能骗了你。

---

## 1. 回测是什么，为什么它是一切量化策略的照妖镜

### 1.1 定义

**回测（Backtesting）** 是在历史数据上模拟交易策略的执行过程，以评估该策略在过去的表现。它的输入是**策略规则 + 历史数据**，输出是**绩效指标**（收益、风险、换手率等）。

### 1.2 回测在量化工作流中的位置

```
因子发现 → 信号生成 → 回测验证 → 参数优化 → 纸交易(Paper Trading) → 实盘(Live Trading)
   ↑______________________________________________↓
              反馈循环（Feedback Loop）
```

回测是**策略从想法到实盘之间最重要的过滤器**。好的回测能替你省下真金白银的亏损；坏的回测则是所有量化惨案的源头——**回测曲线漂亮，实盘一地鸡毛**。

### 1.3 回测的本质矛盾

回测面临一个根本性的张力：

| 维度 | 越接近实盘 | 但代价是 |
|------|-----------|---------|
| **数据频率** | 越高频（tick → 分钟 → 日线） | 数据量爆炸、计算时间暴增 |
| **市场微观结构** | 模拟订单簿、对手方 | 模型复杂度指数级上升 |
| **交易成本** | 手续费 + 滑点 + 市场冲击 | 参数难以精确估计 |
| **约束条件** | 仓位上限、风控限制 | 优化空间收窄 |

> **核心原则：回测的目标不是让曲线看起来最好，而是让曲线最接近实盘。** 如果你的回测 Sharpe 是 5.0 而实盘是 0.5，回测的价值为零——甚至为负，因为它给了你虚假的信心。

## 2. 回测的两种范式

### 2.1 向量化回测 (Vectorized)

对整个时间序列一次性计算，用矩阵/向量运算。

```
信号 = 计算因子(历史OHLCV)      # [T x 1] 向量
持仓 = 信号转仓位(信号)          # [T x 1]
收益 = 持仓 * 价格变化            # [T x 1] 逐元素
累计PnL = cumsum(收益)
```

| 优点 | 缺点 |
|------|------|
| 极快（numpy 向量化） | 无法建模路径依赖 |
| 代码简洁 | 难以处理限价单/部分成交 |
| 适合因子研究阶段 | 假设流动性无限 |

**适用场景**: run4 的因子质量评估（只需算 IC/ICIR/R²，不需要精确回测）。

### 2.2 事件驱动回测 (Event-Driven)

按时间顺序逐分钟/逐笔模拟：数据到达 → 生成信号 → 下单 → 撮合 → 更新持仓。

```
for t in timestamps:
    bar = data[t]                    # 新数据到达
    signal = strategy.on_bar(bar)    # 策略生成信号
    orders = signal_to_orders(signal) # 信号转订单
    fills = matcher.match(orders, bar) # 撮合引擎
    portfolio.update(fills)           # 更新持仓
    recorder.log(t, portfolio)        # 记录状态
```

| 优点 | 缺点 |
|------|------|
| 真实模拟交易流程 | 实现复杂 |
| 支持限价单/止损单 | 速度慢（O(n²) 撮合） |
| 可引入冲击模型 | 需要更多参数 |

**适用场景**: run5 的完整策略回测（限价单 + 冲击模型）。

### 2.3 选择建议

| 维度 | **事件驱动（Event-Driven）** | **向量化（Vectorized）** |
|------|--------------------------|----------------------|
| **工作方式** | 按时间顺序逐事件（bar/tick）触发逻辑 | 对整个矩阵做向量运算 |
| **速度** | 慢（Python 纯循环尤甚） | 极快（NumPy/pandas 底层 C） |
| **精度** | 高——可精确模拟订单簿交互 | 低——隐含"按收盘价成交"假设 |
| **适合场景** | HFT 研究、做市策略、需要精确成交逻辑 | 初步筛选、因子研究、低频策略 |
| **典型框架** | Backtrader, QuantConnect(Lean), 自研 C++ | Zipline, VectorBT, 自研 pandas |
| **最大风险** | 实现复杂度过高，bug 概率大 | **前视偏差（Look-ahead Bias）极难避免** |

> **实用建议：向量化用于快速筛选（成千上万个想法的第一轮过滤），事件驱动用于深度验证（进入候选池的策略必须过这关）。**

## 3. 回测框架的核心模块

任何一个严肃的回测框架，都可以解耦为以下六层：

```
┌─────────────────────────────────────────┐
│  6. 绩效评估层（Performance Layer）       │  输出：Sharpe, MDD, PnL Curve...
├─────────────────────────────────────────┤
│  5. 执行模拟层（Execution Layer）        │  输出：成交价、成交量、成本
├─────────────────────────────────────────┤
│  4. 组合构建层（Portfolio Layer）        │  输出：目标持仓向量
├─────────────────────────────────────────┤
│  3. 信号生成层（Signal/Alpha Layer）     │  输出：每只资产的预测收益/信号值
├─────────────────────────────────────────┤
│  2. 数据层（Data Layer）                 │  输出：清洗后的 OHLCV / 基本面数据
├─────────────────────────────────────────┤
│  1. 事件驱动引擎（Event Loop Layer）     │  调度：按时间顺序驱动各层
└─────────────────────────────────────────┘
```

### 3.1 第一层：数据层（Data Layer）

数据层是回测系统最基础、也最容易被忽视的部分。

**数据来源**:

| 数据类型 | 用途 | 频率 | 典型来源 |
|----------|------|------|---------|
| **价格数据（OHLCV）** | 计算收益、执行信号 | 日频/分钟/Tick | 交易所、Polygon.io、Binance |
| **公司基本面** | 计算因子暴露（B/M, ROE） | 季频/年频 | Compustat, Wind |
| **一致预期** | 盈利超预期因子（SUE） | 月频/日频 | IBES, FactSet |
| **公司行为** | 拆股、配股、分红调整 | 事件驱动 | CRSP, Wind |
| **幸存者/退市数据** | 避免幸存者偏差 | 静态 | CRSP delisting returns |

**六个数据陷阱**:

| 陷阱 | 后果 | 解决方案 |
|------|------|---------|
| **幸存者偏差** | 回测收益系统性高估 | 使用 Point-in-Time 数据库 |
| **前视偏差** | 回测收益严重高估 | 严格按公告日期对齐数据 |
| **复权错误** | 收益计算错误 | 使用 Total Return 数据 |
| **停牌处理不当** | 波动率被低估，Sharpe 被高估 | 停牌期间收益设为 NaN |
| **日内时间戳不一致** | 跨市场信号错位 | 统一到 UTC |
| **数据清洗不足** | 信号被噪音淹没 | 建立数据质量仪表盘 |

### 3.2 第二层：信号生成层（Signal/Alpha Layer）

信号（Signal）是对未来收益的条件期望的某种代理。

**信号处理流程**:
```
原始因子值（Raw Factor）
    ↓ 截面标准化（Cross-sectional z-score）
    ↓ 缩尾（Winsorization，如 1%/99% 分位数）
    ↓ 行业中性化 / 市值中性化（可选）
最终信号（Clean Signal）
    ↓ 映射函数（线性 / 非线性）
目标仓位 / 目标权重
```

### 3.3 第三层：组合构建层（Portfolio Layer）

**五种加权方式**:

| 方式 | 公式 / 逻辑 | 优点 | 缺点 |
|------|-----------|------|------|
| **等权** | $w_i = 1/N$ | 简单、分散、稳健 | 忽视信号强弱差异 |
| **信号加权** | $w_i \propto s_i$ | 直接反映信号强弱 | 极端信号占主导 |
| **市值加权** | $w_i \propto MCAP_i$ | 可投资性强 | 信号效果被市值稀释 |
| **市值+信号混合** | $w_i \propto MCAP_i \cdot s_i$ | 兼顾可投资性和信号强度 | 需回测验证 |
| **风险平价/均值方差** | 基于协方差矩阵优化 | 风险预算清晰 | 对协方差估计极度敏感 |

### 3.4 第四层：执行模拟层（Execution Layer）

这是区分"学术回测"和"实战回测"的关键层。

**交易成本的三个组成部分**:
$$
\text{实际成交价} = \text{信号价格} - \text{手续费}(commission) - \text{滑点}(slippage) - \text{市场冲击}(market\ impact)
$$

**市场冲击模型的层次**:

| 层次 | 方法 | 使用场景 |
|------|------|---------|
| L0 固定成本 | `cost = commission + spread_half` | 策略初筛 |
| L1 线性冲击 | Almgren-Chriss 简化版 | 进入候选池 |
| L2 平方根+衰减 | Propagator 模型（MadEvolve 使用） | 深度验证 |
| L3 完整 Propagator | Bouchaud et al. 2004, 2009 | 机构级 |

**MadEvolve 的冲击模型**: 6 参数平方根冲击模型

```
瞬时冲击 = σ * a * (Q / V)^b          # 立刻发生
永久冲击 = σ * c * |Q / V|^d * sign(Q) # 信息泄露
衰减函数 G(τ) = e^(-τ / τ_0)          # 指数衰减
总冲击 = 瞬时冲击 * G(τ) + 永久冲击   # 随时间衰减
```

| 参数 | 含义 | 典型值 |
|------|------|--------|
| σ | 波动率 | 从数据估计 |
| a | 瞬时冲击系数 | ~0.1 |
| b | 瞬时冲击指数 | ~0.5 (平方根定律) |
| c | 永久冲击系数 | ~0.01 |
| d | 永久冲击指数 | ~0.5 |
| τ_0 | 衰减时间常数 | ~1 分钟 |

**我的经验**: 冲击模型的计算是 O(n²) 的（每笔交易要累加之前所有交易衰减后的冲击）。MadEvolve 复现中用 numpy 向量化 + 预计算衰减查表 + 滑动窗口裁剪，把 42 分钟压缩到 7 分钟。

### 3.5 第五层：绩效评估层（Performance Layer）

**核心指标**:

| 指标 | 公式 | 含义 | 健康范围 |
|------|------|------|---------|
| PnL | Σ(成交收益 - 手续费 - 冲击成本) | 总盈亏 | — |
| 年化收益 | $(1 + R_{total})^{252/T} - 1$ | 标准化的盈利能力 | — |
| Sharpe | $(R_{ann} - R_f) / \sigma_{ann}$ | 风险调整收益 | > 0.5 可用，> 1.0 好 |
| Calmar | 年化收益 / 最大回撤 | 回撤调整收益 | > 1.0 优秀 |
| Sortino | $(R_{ann} - R_f) / \sigma_{downside}$ | 只惩罚下方波动 | — |
| 最大回撤 | $\max(峰值 - 谷值) / 峰值$ | 最坏情况亏损 | < 20% 可接受 |
| 胜率 | 盈利次数 / 总交易次数 | 交易胜率 | 别只看这个 |
| 盈亏比 | 总盈利 / 总亏损 | 每单位亏损能赚多少 | > 1.5 不错 |
| 换手率 | 月度买入总额 / 平均组合净值 | 交易活跃度 | 10%–30% |

**为什么 MadEvolve 选 PnL 而非 Sharpe 作为适应度**:

Sharpe 可以通过"选择性交易"（只在确定性高时交易，减少波动率分母）来人为刷高。冲击调整 PnL 内含成本惩罚——做大单会被冲击成本惩罚，做高频会被手续费惩罚。这是更诚实的适应度函数。

## 4. 回测的七大陷阱

### 陷阱 1：前视偏差（Look-ahead Bias）

**症状**：回测 Sharpe 3.0，实盘 0.3。

**常见形式**：
- 用"修正后"的财报数据做回测（实际公告在 3 个月后）
- 用月底的指数成分股来做月初的交易
- 用全样本做标准化（z-score 用了未来才有的均值和标准差）

**检测**：把所有特征再滞后 1-2 个周期，表现是否大幅下降？

### 陷阱 2：幸存者偏差（Survivorship Bias）

回测中只有今天还活着的公司——破产、退市、被收购的"失败者"不见了。量级：美股中幸存者偏差会使回测收益高估 **1%–2% 年化**。

### 陷阱 3：过拟合与数据窥探（Overfitting & Data Snooping）

你在同一个历史数据上测试了太多假设，最终发现的那个"最好的"只是随机噪音中的极值。

**检测方法**:
- Harvey 标准：用 t > 3.0（而非 1.96）作为显著性阈值
- **Deflated Sharpe Ratio**（Bailey & López de Prado 2014）
- **PBO（Probability of Backtest Overfitting）**（Bailey et al. 2017）
- 记录失败策略的数量——试了 200 个想法只找到 2 个"好"的，这个比例本身就是过拟合的预警

### 陷阱 4：时间序列的特殊性

**致命错误**：对时间序列做随机 K-Fold 交叉验证。

**为什么错**：金融数据是时序依赖的、非平稳的。随机打乱意味着用未来的数据训练过去——终极版的前视偏差。

**正确做法**：推进式交叉验证（Walk-Forward / Purged K-Fold）：
```
Train₁ │ Val₁ │ Test (不可见)
─────────────────────────────────→ 时间
        Train₂ │ Val₂ │ Test
```

### 陷阱 5：交易成本被低估或忽略

**关键测试**：对交易成本做 **2× 压力测试**——如果你的策略在成本翻倍后还赚钱，那才是真正稳健的。

### 陷阱 6：基准的选择性失明（Benchmark Amnesia）

策略年化收益 15%，看起来不错。但同期大盘涨了 20%——你只是承担了市场风险。

### 陷阱 7：回测周期选择偏差

策略在 2009–2019 年（美股最长牛市）表现极好。必须覆盖**至少一个完整牛熊周期**。

## 5. 回测 → 实盘的距离

| 回测假设 | 实盘现实 |
|----------|----------|
| 历史数据完整准确 | 数据有缺失、错误、延迟 |
| 订单立即成交 | 网络延迟、撮合排队 |
| 无对手方风险 | 交易所可能宕机/限频 |
| 无监管限制 | 持仓上限、KYC、税务 |
| 资金无限可分 | 最小下单量、最小价格变动 |

**纸交易（Paper Trading）的必要性**：
- 使用与实盘完全相同的数据源和接口
- 模拟真实的延迟（故意加入 1–3 秒）
- 纸交易至少跑 1-3 个月（覆盖至少一次重大市场事件）

## 6. 开源回测框架对比

| 框架 | 架构 | 适合场景 | 优点 | 缺点 |
|------|------|---------|------|------|
| **Backtrader** | 事件驱动 | 中低频策略，教学 | 文档详细，社区大 | 速度慢 |
| **Zipline-Reloaded** | 混合 | 日频因子策略 | Pipeline API 优雅 | 社区几乎停滞 |
| **VectorBT** | 向量化 | 超快速筛选 | 极快（100×+） | 不支持复杂逻辑 |
| **QuantConnect (LEAN)** | 事件驱动 | 全频段，多资产 | 最完整开源引擎 | 学习曲线陡峭 |
| **Qlib (微软)** | 向量化+ML | 日频 AI 选股 | 全流程 | 偏学术/AI |
| **VN.PY** | 事件驱动 | A 股/期货实盘 | 国内最成熟 | 回测精度有争议 |

| 你的情况 | 推荐 |
|----------|------|
| 刚入门，想学习回测原理 | **Backtrader** |
| 有 1000 个因子要快速筛选 | **VectorBT** |
| 做一个真正会上实盘的策略 | **QuantConnect (LEAN)** |
| 想完全掌控每一行代码 | **自研** |

## 7. 我的实践经验

从 MadEvolve 复现中总结的回测开发流程：

1. **先跑通向量化版本**——验证策略逻辑，秒级出结果
2. **加入交易成本**——手续费 + 简单滑点模型
3. **换事件驱动框架**——模拟逐分钟决策
4. **加入冲击模型**——评估大仓位策略的真实性
5. **性能优化**——向量化、查表、窗口裁剪
6. **交叉验证**——不同时间段、不同参数、不同随机种子

**一个实用技巧**: 写回测时先不做优化，让代码清晰可读。跑通后再做性能优化。正确的慢回测 > 错误但快的回测。

## 8. 回测设计检查清单

### 数据完整性
- [ ] 数据包含已退市股票（无幸存者偏差）
- [ ] 公司行为（拆股、分红）已正确复权
- [ ] 财务数据按 Point-in-Time 对齐（滞后至少 3 个月）

### 信号生成
- [ ] 所有特征在信号时刻都"可知"（无前视偏差）
- [ ] 中性化/标准化/缩尾仅使用当时可得的数据

### 执行假设
- [ ] 交易成本（手续费 + 滑点 + 市场冲击）已扣除
- [ ] 涨跌停 / 停牌股票已正确处理
- [ ] 我用 2× 成本做了压力测试

### 过拟合防御
- [ ] Train/Val/Test 严格按时间分割（不是随机打乱）
- [ ] 测试集至今只被使用了一次
- [ ] 子周期分析（牛市/熊市/震荡）表现合理

### 基准对比
- [ ] 策略跑赢了同期市场指数
- [ ] Fama-French 归因显示存在显著 Alpha

## 9. 常见误区避坑

| ❌ 常见误区 | ✅ 正解 |
|------------|--------|
| "Sharpe 是唯一需要看的指标" | 同时看 MDD、Calmar、Sortino、冲击调整 PnL |
| "我用了样本外数据，所以策略一定有效" | 记录总测试次数，用 DSR |
| "回测曲线越平滑越好" | 过度平滑可能意味着前视信息 |
| "机器学习模型的回测跟传统因子一样" | ML 需要推进式验证（Walk-Forward） |
| "测试集效果好，万事大吉" | 看子周期和不同市场环境 |
